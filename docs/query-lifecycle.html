How MAGENTRA handles a query

A traced walkthrough of one user message — keystroke to rendered answer — and every point along the way where the agent can stop, stall, or refuse. Line references are from the working tree at ad544db.

Four processes, one pipe. The Electron renderer talks to the Electron main process over IPC; main talks to the engine child process over NDJSON on stdio. Click a stage.

One pipe, one consumer The engine's outbound event stream is single-consumer by design — exactly one for await loop drains it for the engine's lifetime (engine.ts:208). Everything the UI knows arrives through that one ordered channel.

The agent loop

Everything happens inside one unbounded for loop at session.ts:1023. Each pass makes exactly one provider call. The loop only exits through the conditions below.

Guards before the model is even called — capped runs only

Interactive root turns run uncapped: maxIterationsPerTurn and maxTokensPerTurn default to MAX_SAFE_INTEGER, so neither break ever fires. The caps only bite when capped is set — unattended (mission) runs and child sessions. The stall detector, not these caps, is the brake on an interactive turn.

The nudge ladder — what happens when the model stops calling tools

This is the interesting part, and it runs against stopping. When a response comes back with no tool calls, the turn does not simply end — seven checks run in priority order, and the first match pushes a synthetic user message and continues the loop.

:1053 · If a configured Stop hook returns blocked, its reason is injected and the loop continues. Guarded by stopHookFired — at most once per turn.

MAX_AUTO_NUDGES = 3 now caps only the wrap-up nudge The old ceiling that could silence the whole ladder is gone. The signal-driven rungs — length cutoff (rung 2), failed batch (rung 4) and open tasks (rung 5) — now nudge without limit in every turn. MAX_AUTO_NUDGES = 3 (session.ts:72) survives, but the shared nudgeCount counter it guards is spent only by rung 6, the wrap-up nudge (↻ requesting a work summary). Once three wrap-up nudges are spent the turn can end at rung 7 — but a failed tool call or an open task will still keep it going.

OVERDRIVE — same loop, different policy

The turn loop is now identical in both stances. The rungs below run in every turn regardless of OVERDRIVE; toggling OVERDRIVE (/overdrive on or the composer toggle) changes the permission stance and adds cosmetics, not the loop's shape.

Self-verify rung — every turnCONTINUES ONCE

On the first clean end-attempt of any turn, a completeness + economy self-check is injected, judged only against the user's query — no assumed builds or tests. The verify reply is buffered; a bare DONE ends the turn invisibly. Fires once per turn; mid-run steering re-arms it. The ⚡ overdrive: self-verifying… / ✓ overdrive: verified status lines are emitted only when OVERDRIVE is on — the rung runs silently otherwise.

Stall detector — every turnPIVOT → ASK

Three consecutive identical rounds (same tool calls, same results) force a strategy pivot; after two spent pivots the model must ask the user one concrete question. Messages: ⚡ stall detected — forcing strategy pivot N/2 then ⚡ still stalled after pivots — asking the user (STALL_PIVOT_TEXT / STALL_ASK_TEXT). This is the brake that replaces the caps on an interactive turn.

What OVERDRIVE alone changesPOLICY + COSMETICS

The allow-all permission stance; the deletion scope-split (workspace-scoped deletions skip the guard); a pre-turn git stash create snapshot on every root turn; the OVERDRIVE system-prompt section; the status-line chatter; and the frontend identity/cinematic. Mid-run steering is engine-generic — steer_message lands at the next message boundary of the running turn — but the composer only sends it while OVERDRIVE is on.

Failure and interrupt paths

The catch at :1181 repairs dangling tool_use blocks with synthetic tool_results before recording anything — without this the provider rejects the next request and /resume replays the same corruption. The finally at :1215 always clears the busy latch and emits turn_finished, so a crash cannot strand the composer.

Every way a turn can stop

Twelve distinct mechanisms (removed 2026-07-20: plan mode and its approval gate, the three-way permission modes — now a two-state stance, OVERDRIVE off vs on — and the reuse gate's block mode, which is reminder-only now and no longer a stop at all). The distinction that matters most: user-facing stops raise a prompt you can see and answer; silent stops return an error to the model only — from the outside they look like the agent simply refusing or giving up.

Compaction

Runs mid-loop (:1179) and post-turn (:1244). maybeCompact (:1575) self-gates on contextWindow × compactionThreshold (default 0.8) and walks the keep-boundary back so a tool_use is never severed from its tool_result (:1581).

Known historical failure mode A contextWindow value pinned in the global settings layer merges into every workspace. When that value was 4096, compaction looped continuously instead of using the model's real window. The resolver is now single-source via contextWindowFor(model), and the key is documented as an escape hatch for local servers only.

What not to remove The nudge ladder, the dangling-tool_use repair in the catch, and the compaction boundary walk are all load-bearing against stopping. Stripping "planning stops" indiscriminately would take the ladder with it and make premature termination more common, not less. The historically removable levers are all handled now: the reuse gate's block mode is gone (reminder-only), and the iteration/nudge caps no longer apply to interactive turns. What remains adjustable is the plan-first reminder's wording.
